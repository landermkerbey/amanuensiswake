#+TITLE: Deployment Guide — Amanuensis Wake

#+NOTE: This file is public. All server addresses and usernames are intentional
placeholders — do not commit real values. The rsync deploy method described here
is interim scaffolding and will be replaced by a CI/CD pipeline in a
later phase.

* Server setup (run once as root or sudo user)

** Install nginx

#+BEGIN_SRC sh
sudo apt update
sudo apt install -y nginx
#+END_SRC

** Create the site user

#+BEGIN_SRC sh
sudo useradd --system --no-create-home --shell /usr/sbin/nologin amanuensiswake
#+END_SRC

** Create the deploy directory

#+BEGIN_SRC sh
  sudo mkdir -p /var/www/amanuensiswake/dist
  sudo chown -R amanuensiswake:amanuensiswake /var/www/amanuensiswake
  sudo chmod -R 775 /var/www/amanuensiswake
#+END_SRC

** Allow your personal user to deploy

Add your personal user to the =amanuensiswake= group so you can write to the
deploy directory without sudo:

#+BEGIN_SRC sh
sudo usermod -aG amanuensiswake YOUR_USERNAME
sudo chmod -R g+w /var/www/amanuensiswake
#+END_SRC

Log out and back in for the group change to take effect.

** Enable the nginx site

#+BEGIN_SRC sh
sudo cp /path/to/repo/deploy/nginx.conf \
    /etc/nginx/sites-available/amanuensiswake
sudo ln -s /etc/nginx/sites-available/amanuensiswake \
    /etc/nginx/sites-enabled/amanuensiswake
sudo rm -f /etc/nginx/sites-enabled/default
#+END_SRC

** Test and reload nginx

#+BEGIN_SRC sh
sudo nginx -t
sudo systemctl reload nginx
#+END_SRC

** Allow HTTP and HTTPS through the firewall

#+BEGIN_SRC sh
sudo ufw allow 'Nginx Full'
sudo ufw enable
#+END_SRC

* Deploying the site

Run this from your local machine after =npm run build=:

#+BEGIN_SRC sh
  rsync -avz --no-group --delete dist/ \
      amanuensiswake@YOUR_SERVER_IP:/var/www/amanuensiswake/dist/
#+END_SRC

Or if deploying as your personal user (after group setup above):

#+BEGIN_SRC sh
  rsync -avz --no-group --delete dist/ \
      YOUR_USERNAME@YOUR_SERVER_IP:/var/www/amanuensiswake/dist/
#+END_SRC

* Publishing new or updated content

Content lives in the =amanuensiswake-content= submodule at =content/=. The code
repository pins a specific commit SHA from the content repository, so code and
content versions are always auditable together.

** Workflow

#+BEGIN_SRC sh
# 1. In the content repo: commit and push your new or updated .org file
cd content
git add .
git commit -m "feat: add <page-name>"
git push

# 2. Back in the code repo: advance the submodule pointer and push
cd ..
git submodule update --remote content
git add content
git commit -m "chore: update content submodule"
git push

# 3. On the server: pull the new submodule pointer, fetch content, and rebuild
ssh YOUR_USERNAME@YOUR_SERVER_IP
cd ~/amanuensiswake
git pull
git submodule update --recursive
npm run build

# 4. From your local machine: deploy the new build
rsync -rlz --delete dist/ \
    YOUR_USERNAME@YOUR_SERVER_IP:/var/www/amanuensiswake/dist/
#+END_SRC

** Drafts

Keep drafts on a local branch in the content repo. Only merge to =main= when
ready to publish. Never push draft branches to the public remote — the content
repository is public.

* Validating the deployment

#+BEGIN_SRC sh
curl -I http://amanuensiswake.com
#+END_SRC

Expected: =HTTP/1.1 200 OK=

Check clean URLs work:

#+BEGIN_SRC sh
curl -I http://amanuensiswake.com/essays/on-silence
#+END_SRC

Expected: =HTTP/1.1 200 OK= (not 404, not 301)
